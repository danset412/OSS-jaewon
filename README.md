# 오픈소스소프트웨어 개인과제 정리

## 1. 버전 관리 개요

### 버전 관리의 필요성
파일 작업을 하다 보면 과거 상태로 되돌아가야 하는 경우가 자주 발생한다. 기존에는 파일 이름을 바꾸거나 폴더를 복사해 관리했지만, 파일이 많아질수록 비효율적이었다.

버전 관리는 파일의 변경 이력을 체계적으로 기록하여 과거 특정 시점으로 되돌릴 수 있고, 누가 언제 무엇을 수정했는지 확인할 수 있으며, 여러 사용자가 동시에 작업할 때 발생하는 충돌을 관리할 수 있게 해준다.

### 버전 관리란?
시간이 흐르면서 파일이 어떻게 변경되었는지를 기록하는 시스템이다. Git에서는 각 변경 사항을 커밋(commit)이라는 단위로 저장하며, 가장 최신 커밋을 가리키는 포인터를 HEAD라고 한다.

### 버전 관리 도구 방식
- CLI(Command Line Interface): Git Bash 등 명령어 기반 사용
- GUI(Graphical User Interface): Sourcetree 등 그래픽 기반 사용

### 커밋(commit)
커밋은 프로젝트의 현재 상태를 저장하는 스냅샷이다. 변경 사항을 add로 스테이징한 뒤 commit을 통해 저장소에 기록하며, 커밋들이 쌓이며 버전 히스토리를 형성한다.

### 지역 저장소와 원격 저장소
- Local Repository: 내 컴퓨터에 존재하는 저장소
- Remote Repository: GitHub 같은 서버 기반 저장소

주요 명령어는 clone, add, commit, push, pull 이 있다.

## 2. Git과 GitHub 개요

### Git의 특징
Git은 2005년 리누스 토르발스가 개발한 분산 버전 관리 시스템이다. 모든 개발자가 전체 저장소를 보유하므로 빠르고 안정적인 개발이 가능하다.

### Git의 주요 기능
- 파일 변경 이력 추적
- 여러 작업을 병합(merge) 가능
- 브랜치를 활용한 비선형 개발

### Git의 내부 구조
Git은 Working Directory(작업 공간), Staging Area(스테이징 영역), Repository(.git 저장소)로 구성된다. 이 외에도 잠시 작업을 저장하는 Stash 공간이 존재한다.

### 브랜치(branch)
브랜치는 기존 코드를 기준으로 독립적인 작업 공간을 만들어 새로운 기능 개발이나 실험을 안전하게 수행할 수 있는 기능이다.

### GitHub
GitHub는 Git 저장소를 온라인에서 관리할 수 있는 플랫폼으로, 협업 및 오픈소스 개발을 위한 다양한 기능을 제공한다.

## 3. Git 설정과 저장소 생성

### Git 최초 설정
Git 사용 전 한 번만 설정하는 전역 설정은 user.name, user.email, core.autocrlf, core.safecrlf, core.editor, init.defaultBranch 등이 있으며, 설정 정보는 ~/.gitconfig 파일에 저장된다.

### Git 저장소 생성
git init 명령을 사용하여 프로젝트 폴더를 Git 저장소로 초기화할 수 있다. 저장소가 생성되면 .git 폴더가 만들어지며, 이 폴더에는 모든 버전 정보가 저장된다.

---

## 4. 커밋과 로그, 과거 여행
Git은 작업 디렉토리, 스테이징 영역, 저장소의 세 영역을 기준으로 동작한다. git status를 통해 파일 상태를 확인하고 git commit으로 변경 사항을 저장한다.

git log, git show 등을 통해 버전 이력을 확인할 수 있으며, HEAD~, HEAD~2와 같은 표기법으로 이전 커밋을 가리키고 git checkout 명령으로 과거 버전으로 이동할 수 있다.

---

## 5. diff, 파일 삭제와 복원

### diff
diff는 파일의 변경 내용을 비교하여 어떤 부분이 추가되었고 삭제되었는지를 보여주는 명령어로, 코드 리뷰와 버그 수정에 중요한 역할을 한다.

### 파일 삭제 및 복원
rm은 작업 디렉토리에서만 파일을 삭제하며, git rm은 작업 디렉토리와 스테이징 영역에서 동시에 파일을 제거한다. git restore 명령을 사용하면 저장소 상태로 파일을 복원할 수 있다.

---

## 6. 태그와 브랜치

### 태그(tag)
태그는 특정 커밋에 버전 이름을 붙여 관리하는 기능이다. 주석 태그와 일반 태그가 있으며, 중요한 버전을 명확히 구분할 수 있다.

### 브랜치 관리
브랜치는 main을 기준으로 생성되며, 브랜치 이동과 병합을 통해 독립적인 작업을 하나의 코드로 합칠 수 있다. 병합 과정에서 충돌이 발생할 수 있으며, Git은 이를 표시해 개발자가 직접 해결하도록 돕는다.

---

## 7. 원격 저장소 연동

### Clone
git clone 명령을 사용하면 원격 저장소 전체를 지역 저장소로 복제할 수 있으며, origin이라는 기본 별칭이 자동으로 설정된다.

### Push & Pull
push는 지역 저장소의 변경 내용을 원격 저장소에 업로드하는 것이고, pull은 원격 저장소의 변경 내용을 내려받는 것이다. 협업 시에는 pull → 작업 → commit → push 순서를 지키는 것이 중요하다.
